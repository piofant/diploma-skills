# Diploma Format — оформление ВКР под формальные требования ИТМО

Скилл готовит текст ВКР к сдаче по формальным требованиям оформления. Проверять **после конвертации в PDF** — форматирование при `.docx → .pdf` иногда ломается, и проверять надо на той поверхности, которую читает комиссия (если работа в Google Docs — на реальном Google-рендере, а не локальном docx).

Зафиксировано со слов секретаря ГЭК Артёма Петрова (TG, 2026-05-14, чат ВКР К3420/К3421) и научного руководителя Меркушева А.Е. (2026-05-14).

## Структура заголовков и Содержания

- **Третий и далее уровень нумерованного заголовка НЕ выделяется полужирным.** `### 1.1`, `### 1.2` могут быть bold, а `#### 1.1.1`, `##### 1.1.1.1` — **не bold**.
- **Текст СОДЕРЖАНИЯ не выделять полужирным** целиком. Заголовки оглавления — обычным.
- **«Выводы по разделу N» — БЕЗ номера подраздела.** Не `### 1.5 Выводы по главе`, а заголовок уровня раздела (`## Выводы по разделу 1`). Часто проще: вообще не делать «Выводы по разделу», оставить только общее Заключение.

## Нумерация страниц и ссылок

- **Нумерация страниц обязательна** во всём документе (включая Содержание).
- **Нумерация источников — по порядку появления в тексте.** Во Введении номера сначала маленькие, потом большие — никаких прыжков «1, 47, 12».
- **В одном абзаце не нужна повторная ссылка на тот же источник.** Поставил [12] — больше [12] в том же абзаце не ставим.

## Списки сокращений и терминов

- **В разделах «Сокращения» и «Термины» НЕТ красной строки нигде.** Все строки без отступа первой строки.

## Описание базы данных

- **БД описывать через Приложение**, не в основном тексте главы.

## Орфография сложных слов

- **Сложные слова всегда через дефис:** BPMN-диаграмма, WEB-интерфейс, Telegram-бот, AI-сервис, n8n-узел, ER-диаграмма, REST-API.

## Стилистика (фидбек научных руководителей)

- **Безличные конструкции:** «был исследован», «будет рассмотрен», «применяется», «выполнен анализ». 1-е лицо («я взял», «я выбрал», «у меня уходит») недопустимо вне Введения.
- **Запрет на «художественный»/«хабровский» регистр:** «куски», «подвох», «по факту», «по-простому», «вот в чём дело», риторические вопросы — недопустимо. Пример как НЕ надо: «Методология собрана из четырёх кусков». Как надо: «Методология включает четыре составляющих части».
- **Сравнительный анализ — в таблицу, а не в прозу** (Меркушев: «сэкономишь много времени на вычитку»). Параграфы «А — это X. Y делает Z…» заменять детальной таблицей: «Категория | Инструмент | Цена | Функционал | Ограничение». Один большой Markdown-table > 6 однотипных параграфов. (Бонус: таблицы в ИТМО исключены из антиплагиат-проверки.)
- **В заключении — явные баззворды:** «Поставленные задачи выполнены», «цель работы достигнута». Идеально — пронумерованный список «1. Задача N — выполнена: (краткое подтверждение + ссылка на раздел)».

## Чек-лист после конвертации PDF

После сборки `.docx → .pdf` (или после импорта в Google Docs) пройти проверки **на финальной поверхности**:

1. Все страницы пронумерованы (включая Содержание)?
2. Заголовки `####` и глубже — не bold?
3. Содержание не bold?
4. Список сокращений и терминов — без красной строки?
5. Источники в порядке появления в тексте?
6. Все сложные слова через дефис?
7. Описание БД — в Приложении (если есть)?
8. Разделы «Выводы по главе N» убраны (либо переформулированы как `## Выводы по разделу N` без `### N.M`)?
9. Сравнительные обзоры — в таблицах, не в прозе?
10. Заключение — нумерованный список «задача → выполнена»?

## Инженерные грабли build-pipeline (docx/Google Docs)

Выстрадано на реальной сборке ВКР К3420 (pandoc md→docx → Google Docs):

- **Google Docs игнорирует style-level цвет именованных заголовков** и красит их своим синим. Фикс — прямой `<w:color w:val="000000"/>` на каждом run заголовка (не на стиле). Иначе после импорта все заголовки синие.
- **Borderless-таблицы pandoc Google Docs давит** в 1–2-символьные колонки. Нужен явный пост-процессор, проставляющий границы и ширины; и он должен быть **внутри build-скрипта**, а не лежать рядом — иначе следующая пересборка регрессирует.
- **Word `{TOC}` field Google не вычисляет** → Содержание пустое. Нужен статический TOC, собранный из PDF-outline (pypdf), с dot-leader табами.
- **Word `PAGE` field тоже не считается** → нет номеров страниц. Нужен footer с PAGE-field + `<w:titlePg/>` (титул без номера, p2+ с номером).
- **Главный принцип:** фикс-скрипт, который лежит в репозитории, но НЕ вшит в build, — это не фикс, а отложенная регрессия. И проверять — на той поверхности, где работу читают (Google-рендер), а не на локальном docx.
